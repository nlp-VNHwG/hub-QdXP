## 问题1：什么是前后端分离？

### 概念定义
前后端分离是一种软件架构模式，将应用程序的前端（用户界面）和后端（业务逻辑）分离为独立的系统，它们通过API进行通信。

### 06-stock-bi-agent 项目中的实现

#### 后端架构
- **技术栈**：FastAPI
- **核心文件**：
  - `main_server.py` - FastAPI 主服务入口
  - `routers/` 目录 - 包含各种API路由
    - `chat.py` - 聊天相关API
    - `stock.py` - 股票数据API
    - `user.py` - 用户管理API
    - `data.py` - 数据管理API
  - `services/` 目录 - 业务逻辑层
    - `chat.py` - 聊天服务实现
    - `stock.py` - 股票服务实现
    - `user.py` - 用户服务实现
  - `models/` 目录 - 数据模型
    - `orm.py` - 数据库表结构
    - `data_models.py` - API请求/响应模型

#### 前端架构
- **技术栈**：Streamlit
- **核心文件**：
  - `demo/` 目录 - 前端应用
    - `streamlit_demo.py` - 主入口
    - `chat/` 目录 - 聊天功能
      - `chat.py` - 聊天界面
      - `chat_list.py` - 聊天列表管理
    - `user/` 目录 - 用户管理界面
    - `stock/` 目录 - 股票功能界面

#### 通信方式
- 前端通过HTTP请求调用后端API
- 后端返回JSON格式数据
- 前端接收数据后进行展示

#### 优点
1. **职责清晰**：前端专注于用户体验，后端专注于业务逻辑
2. **独立开发**：前后端可以并行开发，提高开发效率
3. **易于维护**：修改前端不影响后端，反之亦然
4. **跨平台**：前端可以适配不同设备，后端保持统一

## 问题2：历史对话如何存储，以及如何将历史对话作为大模型的下一次输入

### 历史对话存储

#### 存储方式
- **数据库**：SQLite
- **存储位置**：`./assert/conversations.db`
- **表结构**：
  - `ChatSessionTable` - 存储会话元数据
    - `id` - 会话ID
    - `user_id` - 用户ID
    - `session_id` - 会话标识（唯一字符串）
    - `title` - 会话标题
    - `start_time` - 会话开始时间
  - `ChatMessageTable` - 存储消息记录
    - `id` - 消息ID
    - `chat_id` - 关联的会话ID
    - `role` - 角色（user/assistant/system）
    - `content` - 消息内容
    - `create_time` - 创建时间

#### 存储流程
1. **初始化会话**：
   ```python
   def init_chat_session(user_name, user_question, session_id, task):
       # 创建会话记录
       chat_session_record = ChatSessionTable(
           user_id=user_id[0],
           session_id=session_id,
           title=user_question,
       )
       # 创建系统消息
       message_recod = ChatMessageTable(
           chat_id=chat_session_record.id,
           role="system",
           content=get_init_message(task)
       )
   ```

2. **追加消息**：
   ```python
   def append_message2db(session_id, role, content):
       # 查询会话ID
       message_recod = session.query(ChatSessionTable.id).filter(
           ChatSessionTable.session_id == session_id
       ).first()
       # 创建消息记录
       message_recod = ChatMessageTable(
           chat_id=message_recod[0],
           role=role,
           content=content
       )
   ```

### 历史对话作为大模型输入

#### 实现方式
- **使用 AdvancedSQLiteSession**：来自 `agents.extensions.memory` 模块
- **核心代码**：
  ```python
  session = AdvancedSQLiteSession(
      session_id=session_id,  # 与系统中的对话id关联
      db_path="./assert/conversations.db",
      create_tables=True
  )
  ```

#### 工作原理
1. **会话关联**：通过 `session_id` 关联到特定的对话历史
2. **历史加载**：从数据库中加载该会话的所有历史消息
3. **上下文注入**：在调用大模型时，自动将历史消息作为上下文注入
4. **流式处理**：使用 `Runner.run_streamed()` 进行流式生成，保持对话的连续性

#### 调用流程
1. **用户发送消息** → 存入数据库
2. **创建会话** → 关联 `session_id`
3. **运行Agent** → Session自动加载历史上下文
4. **大模型生成回复** → 基于完整对话历史
5. **回复存入数据库** → 完成一次对话循环

### 技术特点
- **持久化存储**：对话历史保存在SQLite数据库中，支持长期保存
- **状态管理**：通过 `session_id` 维护不同对话的状态
- **上下文理解**：大模型能够基于历史对话进行连贯的回复
- **流式输出**：使用SSE（Server-Sent Events）实现实时对话体验

# routers/chat.py
@router.post("/")
async def chat(req: RequestForChat) -> StreamingResponse:
    async def chat_stream_generator():
        async for chunk in chat_services.chat(
                user_name=req.user_name,
                task=req.task,
                session_id=req.session_id,
                content=req.content,
                tools=req.tools
        ):
            yield chunk
    
    return StreamingResponse(
        content=chat_stream_generator(),
        media_type="text/event-stream"
    )
